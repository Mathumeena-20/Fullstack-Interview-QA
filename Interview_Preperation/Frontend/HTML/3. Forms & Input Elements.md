# ‚úÖ **1. Difference Between GET and POST Form Submission**

## **GET**

* Data sent in **URL query string**
* Limited data size
* **Not secure** (visible in URL)
* Can be **bookmarked** and **cached**
* Used for **reading/fetching data**

### Example:

```html
<form action="/search" method="GET">
  <input type="text" name="query">
  <button type="submit">Search</button>
</form>
```

URL becomes:
`/search?query=laptop`

---

## **POST**

* Data sent in **request body**
* **More secure** (not visible in URL)
* Can send **large data**
* Cannot be cached or bookmarked
* Used for **creating/updating sensitive data**

### Example:

```html
<form action="/add-user" method="POST">
  <input type="text" name="username">
  <button type="submit">Submit</button>
</form>
```

---

## ‚úî Interview Summary:

* **GET** ‚Üí Fetch data, not secure, URL visible.
* **POST** ‚Üí Send sensitive data, secure, no size limit.

# ‚úÖ **2. Purpose of the `novalidate` Attribute in Forms**

`novalidate` tells the browser **NOT** to run HTML5 validation.

Useful when:

* You want to **handle validation with JavaScript**
* You want to **submit even with empty fields**
* Backend does validation instead of browser

### Example:

```html
<form action="/submit" method="POST" novalidate>
  <input type="email" required>
  <button type="submit">Send</button>
</form>
```

Here, the form will submit **even if email is invalid**.

# ‚úÖ **3. Difference between `<input type="text">` vs `<input type="search">`**

| Feature         | `text`           | `search`                                         |
| --------------- | ---------------- | ------------------------------------------------ |
| General use     | Accept text      | Specifically for search input                    |
| Browser UI      | Plain text field | May show clear (√ó) button                        |
| Mobile keyboard | Normal keyboard  | Search-optimized keyboard (with ‚ÄúSearch‚Äù button) |
| Accessibility   | General purpose  | Identified as search field                       |

### Example:

```html
<input type="text" placeholder="Enter your name">

<input type="search" placeholder="Search products">
```

üí° Functionally similar, but `search` gives better **UX on mobile and for screen readers**.

# ‚úÖ **4. Use of `readonly`, `disabled`, and `required`**

## **readonly**

* Field cannot be edited
* Field **WILL be submitted** with the form

Example:

```html
<input type="text" value="John" readonly>
```

---

## **disabled**

* Field cannot be edited
* Field **WILL NOT be submitted**
* Field appears ‚Äúgreyed out‚Äù

Example:

```html
<input type="text" value="John" disabled>
```

---

## **required**

* Browser will **block form submission** if empty
* Used for mandatory fields

Example:

```html
<input type="email" required>
```

---

| **disabled** | ‚ùå No      | ‚ùå No       | Temporarily inactive fields      |

# ‚úÖ **5. How HTML5 Form Validation Works**

HTML5 provides **built-in validation** without JavaScript.

### Browser checks based on:

* `required`
* `pattern`
* `type="email"`, `type="number"`, `type="url"`
* `min`, `max`, `maxlength`, etc.

If any rule is broken ‚Üí the browser shows an **error message** and **blocks submission**.

---

### Example Form:

```html
<form>
  <input type="email" required>
  <input type="text" pattern="[A-Za-z]{3,}" placeholder="Min 3 letters" required>
  <input type="number" min="1" max="10">
  <button type="submit">Submit</button>
</form>
```

### How the validation works:

1. Browser checks field types
2. If required fields are empty ‚Üí **error popup**
3. If pattern does not match ‚Üí **error popup**
4. If all good ‚Üí form submits

---

### Custom Error Message:

```javascript
input.oninvalid = function () {
  this.setCustomValidity("Please enter at least 3 letters");
};
input.oninput = function () {
  this.setCustomValidity("");
};
```
