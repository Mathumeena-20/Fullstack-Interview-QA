# ================================

# ‚úÖ **1. Difference between Development and Production builds**

# ================================

React produces **two types of builds**:

---

## ‚≠ê Development Build

Used during local development.

### Features:

* **Unminified** (large file size)
* Includes **helpful warnings**
* Includes **error overlays**
* Includes **extra checks (Strict Mode)**
* Slower performance
* Source maps enabled

### Example:

```
react.development.js
```

---

## ‚≠ê Production Build

Used for deployment.

### Features:

* **Minified** (small bundle size)
* Removes all development-only code:

  * warnings
  * prop type checks
  * dev logs
* Faster execution
* No extra safety checks

### Example:

```
react.production.min.js
```

---

### üéØ Interview summary:

> Development build is slow but developer-friendly.
> Production build removes dev warnings, minifies code, and is optimized for speed.

# ‚úÖ **2. What is tree shaking?**

# ================================

Tree shaking = **removing unused code** from the final bundle.

React and modern bundlers use **ES Modules (import/export)** to analyze which parts of code are used.

### Example:

```js
// utils.js
export function add(a, b) {}
export function subtract(a, b) {}
```

If you only import:

```js
import { add } from "./utils";
```

Bundle **removes subtract()** automatically.

---

### ‚≠ê Benefits:

‚úî Smaller bundle size
‚úî Faster load time
‚úî Removes dead code

---

### üéØ Interview summary:

> Tree shaking eliminates unused exports by using static analysis of ES modules.

# ‚úÖ **3. What is Webpack and how does it bundle React code?**

# ================================

Webpack is a **module bundler** that processes React files and bundles them into optimized JavaScript, CSS, images, etc.

---

## ‚≠ê How Webpack works:

### 1Ô∏è‚É£ Entry

Starting point of the application.

```js
entry: "./src/index.js"
```

---

### 2Ô∏è‚É£ Loaders

Transform file types (JSX, CSS, images).

Example:

```js
module: {
  rules: [
    { test: /\.jsx?$/, loader: "babel-loader" },
    { test: /\.css$/, use: ["style-loader", "css-loader"] }
  ]
}
```

---

### 3Ô∏è‚É£ Plugins

Enhance bundling (minification, HTML template, etc.).

Example:

```js
plugins: [new HtmlWebpackPlugin()]
```

---

### 4Ô∏è‚É£ Output

Where the final bundle goes.

```js
output: {
  filename: "bundle.js",
  path: __dirname + "/dist"
}
```

---

### 5Ô∏è‚É£ DevServer

Runs local development server:

```js
devServer: { hot: true }
```

---

## ‚≠ê How bundles are created:

1. Webpack reads `index.js`
2. Follows imports to create dependency graph
3. Uses loaders to process JSX, CSS, images
4. Uses plugins to optimize
5. Produces final bundle used by the browser

---

### üéØ Interview summary:

> Webpack builds a dependency graph of all modules, transforms them using loaders, and outputs optimized bundles for React apps.

# ‚úÖ **4. What is environment variable handling in React?**

# ================================

React apps use environment variables for:

* API URLs
* Environment config
* Feature toggles

### ‚≠ê Naming rule in Create-React-App:

Environment variables MUST start with:

```
REACT_APP_
```

Example `.env` file:

```
REACT_APP_API_URL=https://api.example.com
REACT_APP_VERSION=1.2.0
```

---

### Use in code:

```js
console.log(process.env.REACT_APP_API_URL);
```

---

## ‚≠ê Different `.env` files:

| File               | Purpose                  |
| ------------------ | ------------------------ |
| `.env.development` | Dev environment config   |
| `.env.production`  | Prod environment config  |
| `.env.local`       | Overrides local machines |
| `.env.test`        | Used during Jest testing |

---

## ‚≠ê Why use environment variables?

* Avoid hardcoding API URLs
* Switch dev/prod endpoints automatically
* Hide sensitive config (partially‚Äînot fully secure)

---

### üéØ Interview summary:

> React reads environment variables prefixed with `REACT_APP_`, allowing different configs for dev, prod, and test builds.
